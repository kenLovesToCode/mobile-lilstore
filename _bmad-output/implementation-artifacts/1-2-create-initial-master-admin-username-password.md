# Story 1.2: Create Initial Master Admin (Username + Password)

Status: done

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As an admin,  
I want to create the initial master admin account,  
so that I can unlock admin features on the device.

## Acceptance Criteria

1. **Given** no admin exists  
   **When** I submit a valid username and password  
   **Then** the system creates the master admin and persists it locally (FR2)  
   **And** the password is not stored in plaintext (stored as a secure derived value).
2. **Given** one or more admins already exist  
   **When** I attempt to access any admin creation flow while not authenticated  
   **Then** the system prevents it (no public registration) (FR3)  
   **And** I am directed to admin login instead.
3. **Given** master admin creation succeeded  
   **When** I close and relaunch the app  
   **Then** I am not prompted to create a master admin again.

## Dev Notes

### Developer Context Summary

- Story 1.1 already delivered route-gating shells, DB bootstrap scaffolding, and entry-gate retry/error behavior. Story 1.2 should build directly on that baseline, not replace it.
- The key failure mode to prevent is insecure credential handling. Passwords must never be persisted or logged in plaintext.
- Keep scope constrained: initial master-admin creation only. Full authenticated admin session behavior belongs primarily to Story 1.3.

### Technical Requirements

- Preserve current first-run gating contract:
  - `hasAnyAdmin() === false` -> create flow is visible.
  - `hasAnyAdmin() === true` -> create flow redirects to login.
- Submit path must:
  - Validate form inputs.
  - Derive secure password value using KDF + random salt.
  - Insert exactly one admin record when count is zero.
  - Return a controlled user-safe error when admin already exists.
- Security requirements:
  - Never store plaintext password.
  - Never log password input.
  - Never expose raw internal DB/KDF errors to UI.
- Post-success behavior:
  - Mark admin existence in gate snapshot runtime.
  - Redirect to login immediately.
  - Relaunch must route to login (not create-admin).

### Architecture Compliance

- Keep DB writes in domain services (`src/domain/services/**`), not in React screen components.
- Keep persistence concerns in DB layer (`src/db/**`, migrations in `src/db/migrations/**`).
- Use integer epoch milliseconds for timestamp columns (`created_at_ms`, `updated_at_ms`).
- Continue existing error-handling pattern used by entry gate: user-safe messages + internal console warning.
- Avoid introducing new route groups or public registration routes outside the gated flow.

### Library & Framework Requirements

- Use Expo-managed dependency install for SDK compatibility (`npx expo install expo-crypto`).
- Use `scrypt-js` for password KDF (architecture-selected KDF).
